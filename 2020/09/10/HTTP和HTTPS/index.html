HTTP和HTTPS

HTTP

HTTP 由于是明文传输，主要存在三大风险：窃听风险、篡改风险、冒充风险。
安全的通信需要包括以下四个原则: 机密性、完整性，身份认证和不可否认。
不可否认：不否认已经发生的行为。

HTTPS

HTTPS 最终采用对称加密形式。

对称加密具有加解密速度快，性能高的特点，也是 HTTPS 最终采用的加密形式。

关键问题 通信双方如何传递这一把密钥，因为在传递过程中也会遇到三大风险。

采用非对称加密进行密钥协商传递

对于 server 来说，保管好私钥，发布公钥给其他 client, 其他 client 只要把内容—对称加密的密钥—加密传给 server 即可。

关键问题 server 怎么把公钥安全地传输给 client 呢。如果直接传公钥，也会存在被中间人调包的风险

数字证书 第三方权威机构（Certificate Authority，简称 CA）颁发，在证书中附上公钥，将证书传给client.

问题一： 如何验证证书的真实性，如何防止证书被篡改

使用摘要算法（如 MD5）将证书明文生成摘要，然后再用第三方权威机构的私钥对生成的摘要进行加密得到签名，将签名附在证书后面

客户端拿到证书后也用同样的摘要算法对证书明文计算摘要，用权威机构的公钥解密签名得到摘要，两者对比可以发现报文是否被篡改

CA 公钥如何安全地传输到 client ？

公钥是存在于 CA 证书上，而此证书（也称 Root CA 证书）被操作系统信任

抓包工具中间人能抓取 HTTPS 包的前提是server信任它们的 CA 证书

问题二、 如何防止证书被调包
每个人都可以申请证书，但是证书内容很多，包括域名，过期时间，序列号等等，都会验证。

根证书自己证明自己，我们必须信任它，不然证书信任链是走不下去的（这个根证书内置在操作系统）

如果站点申请的是二级 CA 颁发的证书，client 收到之后会如何验证这个证书呢？

service 除了传给二级 CA 的证书外，还会把证书信任链也一起传给客户端。客户端根据证书信任链从根证书一级一级解析出密钥。

银行问题，双向认证
客户端也需要证书。如U盾。

关于Cookie的可获取路径问题(cookie.setPath("/");)

正常的cookie只能在一个应用中共享，即一个cookie只能由创建它的应用获得。
有多条cookie.setPath("XXX");语句的时候，起作用的以最后一条为准。
具体说明则是：
当Cookie的path为null时候，则Cookie只能在创建的URI路径下获取到。
当Cookie的path设置值不为null的时候，以设置的URI值为准。

对于URI的匹配规则如下：

  1. 当URI的path值是以“/”结尾的时候，直接设置为cookie的path值
  2. 当URI的path值不是以“/”结尾的时候，查看path里面是否有“/”
    （1）. 如果有“/”的话，直接截取到最后一个“/”，然后设置为cookie的path值。
    （2）. 如果没有“/”的话，将Cookie的path设置为”/”。
    例如: 设置cookies的路径是/user/login,按照截取规则，Cookie可以在/user/*获取到。
