git操作

git三个分区


三个分区之间的转换通过以上命令进行。工作区就是我们当前的文件目录。

常用命令

git remote add 仓库命名 git@192.168.1.119:ndshow(仓库地址)

git config --global user.name "xxx" # 配置用户名
git config --global user.email "xxx@xxx.com" # 配置邮件

git config --global user.password 'xxx' # 配置密码

git branch -m oldName newName # 重命名本地分支

git push --delete origin oldName # 删除远程分支

.gitignore只能忽略那些原来没有被track的文件，如果某些文件已经被纳入了版本管理中，则修改.gitignore是无效的，解决方法
git rm -r --cached . 清空暂存区，重新添加。

git diff

git diff 不加任何参数，将工作区（未add的内容）和暂存区进行比较；
git diff HEAD，将工作区与HEAD指针指向的commit进行比较，一般来说我们当前的改动就是在HEAD指向的commit的基础上进行改动；
git diff --cached，将暂存区与仓库当前commit进行比较；
git diff dev，将工作区与目标分支的最新commit进行比较
git diff [commitId_1] [commitId_2]，将两个commit（id写前4位）进行比较

git commit

commit会在暂存区生成快照，然后推到本地仓库,commit是在HEAD指向的提交上修改。
git commit --amend，用新提交覆盖前一个提交。旧commit则被抛弃掉
然后使用git push --force-with-lease origin main推送到远程仓库
GitHub目前不支持git commit --amend去追加修改而保持不增加提交记录。

git checkout

git checkout [branch_name]切换分支，HEAD会指向指定的分支或者commit,会把仓库中的内容复制到暂存区和工作区中，覆盖之前的版本，而只存在于之前版本的文件则会被删除。

git reset 使HEAD指向指定的commit.

git reset [commitId] --sort，这是最弱的回滚方式，只改变commit信息，不影响暂存区和工作区；
git reset [commitId]，不携带参数时，默认只回滚暂存区，也就是把指定提交的信息复制到暂存区，但是不影响工作区；
git reset [commitId] --hard，这种方式将指定commit回滚到工作区和暂存区。

合并

git merge [branch] 合并（将指定分支合并当前分支，注意和git rebase的区别）有很多中方式，默认会选择最合适的方式，合并是对仓库进行操作。
git merge -s [策略名称] 指定合并策略。

Fast-forward 是最简单的一种合并策略，并两个没有分叉的分支时的默认行为。
Recursive 是Git在合并两个有分叉的分支时的默认行为。三路合并。

三路合并：将当前分支最新commit和指定分支最新commit合并，是
1.将当前分支最新commit（A）和指定分支最新commit（B），以及两个分叉分支的最近公共祖先commit（Base）合并；
2.然后将合并结果拷贝到暂存区和工作区；
3.然后再产生新的提交，提交的祖先为指定分支和当前分支。

为什么三路合并？（A和B两个分支最新提交，到底是A修改了B还是B修改了A？因此需要三路合并）

比较：如果A和Base内容相同，那么说明B修改了，因此合并后是B。

冲突—如果A和Base和B都不一样怎么办？只能手动合并（自己的代码就出现了这个问题）

递归三路合并：Git的分支往往非常繁杂，会导致合并A/B时，能找到多个A/B的共同祖先，而所谓的递归三路合并就是，对它们的共同祖先继续找共同祖先，直到找到唯一一个共同祖先为止，这样可以减少冲突的概率。

git rebase [xxx] 表示将当前分支提交到指定xxx分支上。(注意是当前分支到xxx上，不是xxx到当前分支上)
例子：

git rebase master：
1.将当前分支从最近祖先节点开始的所有提交连接到master后。
2.出现冲突，手动解决冲突，
3.解决后git add .
4.再git rebase --continue ，所有提交的commitId改变了。同时，旧分支上的节点因为没有了引用则会被丢弃，HEAD指向最新的提交。
